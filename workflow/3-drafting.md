# 3-drafting

工作流 3：正文写作（法律解释方法贯穿全程）

目的：根据用户已确认的解题大纲，按"内在结构 + 比例原则 + 鉴定式语言 + 法律解释方法"展开各法律关系对的详细分析，输出 Markdown 草稿。

核心写作哲学：法律解释方法（文义/体系/历史/目的/合宪/比较法）不是"找不到权威依据时的兜底"，而是贯穿于论证全程的增强器。即便已经从用户 KB、案例、学术论文中获得了观点素材，也仍要用解释方法对这些素材进行检验、整合、显性化，借此提升论证的可信度与说服力。

## 〇、W3 启动门禁（三重前置门禁 · 强制）

本门禁是落笔前的硬阻断——不通过本门禁不得开始撰写正文段落。门禁未通过即直接落笔，是被反复验证的格式漂移根因。

双权威源架构：
- 结构骨架（动态）：W2 闸一确认请求权后从解题大纲生成引擎（methodology/outline-engine/）构建检视程式
- 格式规范（静态）：已内嵌于 canonical-report-template.md §三 9 级标题矩阵

大纲填充骨架：侵权五分法 / 违约四分法 / 物权返还程式 / 不当得利程式

### 0.1 三重前置门禁必读

W3 启动前 subagent / 主线程必须完整 Read 以下三份文件：
1. `examples/canonical-report-template.md` —— 格式唯一可信源（9 级标题矩阵 + 大纲填充骨架 + 14 项 checklist + 大纲/正文关系示范）
2. `format/output-format-spec.md` —— 字体 / 字号 / 行距 / 转换工具规范
3. W2 闸二已确认的请求权对应的解题大纲生成引擎或用户知识库（如有）取得检视程式片段（必须实调，不得凭模型记忆）

### 0.2 14 项格式 checklist 自勾

落笔前在 workspace 输出一个 `format-precheck.md` 文件，逐项勾选 canonical-report-template.md §九 的 14 项 checklist。任一项未勾即视为门禁未通过，须先在大纲层补全后再落笔。

### 0.3 不通过则回退

- checklist 任一项未勾 → 回到 W2 闸二补大纲层级标注，再启动 W3
- 解题大纲生成引擎构建的检视程式与案件类型明显不一致 → 回到 W2 闸一重新确认请求权类型
- canonical-template §四 的骨架均不能完美覆盖本案 → 回到 W2 闸二与用户对齐自定义骨架

## 一、写作单元的划分

每个"法律关系对"独立为一个写作单元。

- 法律关系对 = 1 → 主流程顺序写作；
- 法律关系对 ≥ 2 → 启动多 subagent 并行写作。

## 二、并行写作的协议（≥2 法律关系对时）

### 2.1 subagent 统一前提

每个 subagent 的 prompt 必须包含以下统一前提（来自 W2 闸二已拍板的最终大纲）：

- 所有权归属结论：本案中由 W2 分析确定的物权链路终点；
- 各当事人确定身份：非推测性表述，直接使用 W1 确认的当事人表；
- 引用格式规范：《法规名》第 XX 条（有书名号+空格）；司法解释用已定义简称。

### 2.2 启动 subagent

为每个法律关系对启动一个 general-purpose subagent，传入：
1. 案情提要（统一文本）；
2. 该法律关系对所对应的设问；
3. 该部分大纲（含完整要件树和法规括注）；
4. 写作规范文件路径：gutachten-writing-rules.md、proportionality.md、legal-interpretation-toolkit.md、inner-structure.md、examination-sequence.md；
5. 共享上下文：当事人称谓表、术语统一表（避免各 subagent 的不一致）；
6. §2.1 统一前提（所有权归属、当事人身份、引用格式）；
7. 输出文件路径：`draft/<part>.md`。

每个 subagent 自行完成：
- 调用解题大纲生成引擎获取检视程式，通过 `[SLOT-LEGAL-DB]` 获取法条原文；
- 即时校验（见下文 §2.4）；
- 在涉及解释空间的要件上始终运用法律解释方法（见下文 §3.4），并按"素材分级"调用外部依据；
- 输出符合规范的 Markdown 草稿到指定路径。

### 2.3 主流程合并

所有 subagent 完成后，主流程读取各部分草稿，按大纲顺序拼接。合并时检查：
- 当事人称谓 / 术语 / 缩写是否统一；
- 跨部分引用是否准确（如"第二部分已论证乙的同时履行抗辩成立……"）；
- 脚注编号是否连续；
- 重复内容是否合并或交叉引用。

### 2.4 即时法条校验（写作过程中强制）

subagent 在写作过程中每引入一条新法条，必须：
1. 立即调用 `[SLOT-LEGAL-DB]` 查验（不得事后补验）——先查 `verification-log.md`，已 ✓ 直接复用，否则即时调用；
2. 在引用块后标注 `[✓已校验]`；
3. 将该条文写入 `verification-log.md`（追加模式）；
4. 司法解释类条文：100% 强制校验，零容忍——此类条文 OCR 差错率高，不允许任何复用豁免。

### 2.5 格式统一要求

所有 subagent 输出必须遵守：
- 法规引用：《民法典》第 XX 条（书名号+空格）
- 不使用外语方法论标签（Obersatz 等），使用中文括号标注
- 各部分间用 `---` 分隔
- 标题层级：严格按 canonical-report-template.md §三 9 级标题矩阵

## 三、写作展开顺序（每个写作单元内部）

### 3.1 案情涵摄

每项检视开始前先把"本案事实"映射到大纲所列的请求权要件上。建议用一段过渡性叙述：

"依本案查明事实，甲于 X 年 X 月 X 日与乙签订《Y 合同》，约定……。乙未按约定履行，导致甲损失 Z 元。下文围绕甲对乙的违约损害赔偿请求权，依《民法典》第577条之要件展开检视。"

### 3.2 三层四步逐一展开

按 inner-structure.md 的固定顺序展开"权利产生 → 权利消灭 → 权利行使"三层；每层内"构成要件 → 产生抗辩 / 消灭事由 / 行使阻却"四步走。

### 3.3 比例原则的应用

依 proportionality.md 第二节的标准，对每个要件做"明显 / 争议 / 核心争议"分类，分别采用裁判式 / 鉴定式 / 完整鉴定式 + 法律解释。

### 3.4 法律解释方法的运用（贯穿增强 + 素材分级）

**根本立场**：法律解释方法是论证的增强器（Argumentationsverstärker），不是兜底机制。即便用户 KB、案例、学术资源已能提供观点素材，仍要按工具箱（legal-interpretation-toolkit.md）所列方法对这些素材进行重整、检验与显性化，使每一步涵摄都"既有依据，也有方法"。

#### 3.4.1 何时启动解释（触发条件）

任一条件满足即启动：
- 要件含义为不确定法律概念或一般条款（"重大"/"合理"/"善意"/"公平"/"显失"等），且本案事实落入边缘地带；
- 要件文义在通常理解下能涵盖多种情形，本案事实落在哪种存在歧义；
- 双方当事人对该要件含义存在合理争议；
- 现行司法实践对该要件存在分歧；
- 引用的评注、案例或论文之间存在观点冲突，需要选择并证立其中一种。

不满足上述条件的要件，不启动详细解释流程——按比例原则做裁判式简要涵摄即可。

#### 3.4.2 方法贯穿：始终使用，不因素材充足而省略

无论是否已有外部素材可引，下述步骤都要完成：
- **文义起点**：明示要件用语在法律共同体中的通行含义，划定可能解释的合理外延；
- **体系协调**：在民法典本条所在编/章/节、相邻条文、相对应制度间寻找语义协调和功能分工；
- **历史还原**（如必要）：探寻立法者制定时所追求的具体规整意图与所欲解决的问题；
- **目的支撑**：明示该要件在当下法秩序中所要保护的法益和所欲避免的损害；
- **合宪性 / 比较法校验**（如适用）：检查解释结论是否与宪法基本权利、相同制度的比较法处理相协调。

输出形式上，要让读者能看到方法在工作——不要只丢一个结论。例如：

> 文义上"重大误解"指当事人对法律行为内容的认识有错误且该错误对意思表示具有决定意义。体系上，民法典第147条与第148条（欺诈）、第151条（显失公平）共同构成意思表示瑕疵规则群，故本条所欲规整的应限于"非外部诱因导致的内容认识错误"。从立法目的看，本条系在自治原则与交易安全之间寻求平衡，故"误解之重大性"应同时考虑表意人主观判断的分量与受领人合理信赖的强度。综上……

#### 3.4.3 素材分级（为方法提供论据，而非替代方法）

下列三级素材按可获得性顺序调用，但任何一级素材都不能替代解释方法本身——它们是为解释结论提供论据支撑：

**一级：用户已注册的解释 / 评注类知识库（最权威）**

读取 `.user-config.md` 中的 `## 法律解释知识库注册` 段落。如有：
- 用 Skill 工具调用用户已注册的 KB skill（评注类、司法解释释义类、教义学专著类等）；
- 检索目标条文 / 要件，提取学理观点和释义；
- 运用方法对素材进行检验：评注观点是否经得起体系 / 目的解释的复核？是否符合本案语境？是否需要在通说基础上做扩张 / 限缩？
- 脚注组装（尽力字段 + 兜底字段）：详见 methodology/gutachten-writing-rules.md §四"来自用户私有知识库的引用"；
- 进入涵摄环节。

**二级：案例检索 + 学术联网搜索（补充与佐证）**

无论一级是否命中，本级原则上都要执行——案例和论文用于检验通说在司法实践中的稳定性、识别少数说与新近发展、补充本案特殊语境下的处理思路。
- 通过 `[SLOT-LEGAL-DB]`（用户配置的法律数据库检索工具），按"法条号 + 要件关键词"检索相关案例与裁判规则；
- 通过 `[SLOT-WEB-SEARCH]`（用户配置的全网搜索工具），按"要件名 + 法条号 + 学说 / 释义"检索学术论文与实务文章；
- 综合提炼通说 / 多数意见 / 主流司法观点，并标注分歧之处；
- 运用方法对素材进行整合：案例 / 论文之间的观点差异，应通过文义 / 体系 / 目的解释加以调和或显性选择；
- 在脚注中标注 `[案例]` / `[联网]` 并附 URL 或案号。

**三级：方法论自助式解释（无外部素材时仍要完成解释）**

如一级、二级都未获得权威依据：
- 严格依 methodology/legal-interpretation-toolkit.md 的工具箱完成独立解释（文义 → 体系 → 历史 → 目的 → 合宪 → 比较法位阶）；
- 严格按工具箱第六节"自我审查清单"逐项检查；
- 不加任何脚注（真实来源原则：自助式方法论解释属正文论证密度的内生组成，非"权威素材引用"，故不加脚注；严禁 [模型推理] 占位脚注）；
- 进入涵摄环节。

#### 3.4.4 解释段落的写作呈现

无论素材调用到哪一级，解释段落统一格式：

1. **争点限定**：明确所争论的是哪一构成要件要素或法律后果。
2. **四要素展开**：文义—体系—历史—目的，每一项给出独立小结。
3. **冲突权衡**：当各要素结论不一致，明示权衡过程与赋权理由。
4. **总结论**：得出最终解释结论并涵摄案件事实。
5. **反对说回应**：预设一个最强的反对意见，并在论证中明示其要害所在。

具体写作模板与纪律详见 methodology/legal-interpretation-toolkit.md §五"解释的论证规范"。
