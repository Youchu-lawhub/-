鉴定式民法案例研习超级工作流

本 skill 是一份面向所有用户的、可移植的鉴定式民法案例分析工作流。它将"请求权基础方法（Anspruchsgrundlage）"与"鉴定式写作（Gutachten）"两套方法论编排为四段工作流，内置解题大纲生成引擎（outline-engine）自主完成请求权基础鉴定式分析。

本 skill 为纯净版，不绑定任何特定 Agent 应用环境或 MCP 产品。所有外部工具依赖均通过"能力槽（Capability Slot）"抽象，首次使用时由冷启动访谈引导用户自行配置。

## 一、什么时候用本 skill

用户提交民事案件材料并需要按学术规范输出案例分析报告时——典型场景：

- 法学院学生 / 教师准备案例研习作业；
- 律师 / 法务为复杂案件做"内部全要件检视"作为后续诉状/答辩状/代理意见的底稿；
- 法律研究者撰写案例研究文章；
- 用户希望系统理解一个真实案件的全部请求权 / 抗辩 / 聚合竞合关系。

不适用：起诉状 / 答辩状 / 代理意见等具体诉讼文书（请使用专门的诉讼文书 skill）；快速合规咨询（请使用法律研究 skill）。

## 三、工作流（四段）

```
[首次使用] 0-cold-start-interview → 写入 .user-config.md（能力槽注册 + 偏好设置）

每次会话（严格遵守"事实—设问—规范"阀门，内含于 W1/W2）：

┌─────────────────── W1（事实+设问层） ───────────────────┐
│  1-case-intake    （PDF/Word/图片 → 中性事实笔记 + 设问逐字 + 法律关系对）│
└────────────────────────────────────────────────────────────┘

┌─────────────────── W2（规范层起点 · 两闸独立确认） ───────────────────┐
│  2a-preselection  （解题大纲生成引擎 + 法律数据库双层校验 → 候选清单 → 闸一确认）│
│  2b-outline       （已确认清单展开为三阶段要件大纲 → 闸二确认）               │
└────────────────────────────────────────────────────────────────────────┘

┌─────────────────── W3（并行写作） ───────────────────┐
│  3-drafting       （≥2 法律关系对多 subagent 并行；即时校验；法律解释贯穿）│
└────────────────────────────────────────────────────────────┘

┌─────────────────── W4（差异核验+输出） ───────────────────┐
│  4-verification-and-output （增量法条校验+覆盖忠实+格式扫描 → 一次性修订 → 输出）│
└────────────────────────────────────────────────────────────┘
```

每段独立文件位于 workflow/ 目录下，用户也可单独调用某一段（如已有大纲直接进入 3-drafting）。

**阀门约束**（内含于 W1/W2）：

W1 输出只允许事实层 + 设问逐字 + 法律关系对——禁止条文号、法律定性、争点预判；
W2 是法律层的真正起点——条文号、要件解释、检视程式只能从此段开始，且每条文须经法律数据库双层校验；
跨阀门污染一律视为方法论违例，应回到对应阶段重做。

## 四、能力槽体系（Capability Slots）

本 skill 不绑定任何特定工具产品。所有外部依赖均抽象为"能力槽"，首次使用时由冷启动访谈（0-cold-start-interview）引导用户注册自己已配置的工具。

### 4.1 五个标准能力槽

| 编号 | 能力槽名称 | 槽位标识符 | 功能描述 | 是否必须 |
| --- | --- | --- | --- | --- |
| S1 | 法律数据库检索 | `[SLOT-LEGAL-DB]` | 法条检索、条文原文获取、现行有效性校验、司法解释检索 | 必须（可降级） |
| S2 | 文档解析 | `[SLOT-DOC-PARSE]` | PDF / Word / 图片 → Markdown 转换 | 可选 |
| S3 | 全网搜索 | `[SLOT-WEB-SEARCH]` | 学术论文、实务文章、法律资讯检索 | 可选 |
| S4 | Word 输出 | `[SLOT-WORD-OUTPUT]` | Markdown → Word (.docx) 格式转换与输出 | 可选 |
| S5 | 学术论文检索 | `[SLOT-PAPER-SEARCH]` | 学术论文全文获取（如各类学术论文检索平台 / MCP 等） | 可选 |

每个能力槽记录用户在 `.user-config.md` 中注册的具体工具名称、调用方式（MCP 服务器名 / 工具名 / skill 名称等）以及覆盖范围。工作流执行时，按槽位标识符查找用户注册的实际工具进行调用。

### 4.2 硬依赖（必须 · 可降级）

**S1 法律数据库检索（必须）** —— 实时校验法条号、条文原文、现行有效性的工具。用户在冷启动访谈中注册自己已配置的法律检索工具（如各类法律数据库 MCP、法律检索 API 等）。该能力槽提供动态的"现行有效性"校验，是法条引用铁律的执行工具。未配置时允许降级，但报告中所有法条引用必须全量打标（详见 §六-1）。

**解题大纲生成引擎（methodology/outline-engine/）** —— 本 skill 的思维内核，一套从第一性原理自主完成"找法 → 规范属性识别 → 构建检视程式 → 要件拆解 → 大纲成型"的方法论引擎。本引擎使工作流在无外部知识库时也能精准完成请求权基础鉴定式分析。这是内置组件，不属于外部能力槽。

#### 4.2.1 可选增强（非必须）

**用户知识库（如有）** —— 用户自建的请求权基础检视程式知识库或法律评注类知识库。当此 KB 可用时，在 2a-preselection 与 3-drafting 阶段优先参考其预制检视程式，并与解题大纲生成引擎互相校验。当此 KB 不可用时，解题大纲生成引擎独立完成全部检视程式构建——这是本 skill 脱离外部知识库仍能精准工作的核心保障。

### 4.3 降级策略

当依赖资源不可用时，按以下规则降级而非中止：

| 资源 | 正常路径 | 降级行为 |
| --- | --- | --- |
| S1 法律数据库检索 | 调用用户注册的法律检索工具实时校验 | 条文引用全量标注（⚠ 未校验），报告末尾附"待校验清单" |
| 解题大纲生成引擎文件 | 读取 methodology/outline-engine/ | 基于模型请求权基础方法论知识构建临时检视程式，标注 [方法论自主构建] |
| 用户知识库 | 参考预制检视程式 | 解题大纲生成引擎独立完成（此为设计常态，非降级） |
| S4 Word 输出 | 调用用户注册的 Word 转换工具 | 输出 Markdown 草稿 + 提示用户自行转换 |
| 评注/学术类 KB | 涵摄论证中引用深度评注 | 走法律解释工具箱自助式路径，正文论证密度不得降低 |

降级后的产物须在报告首页或闸确认时向用户明示降级状态与影响范围。

### 4.4 软依赖（可选 · 缺失时优雅降级）

| 能力槽 | 用途 | 缺失时降级方案 |
| --- | --- | --- |
| S3 全网搜索 | 学术论文 / 实务文章检索 | 进入"自助式解释"层，标注 [需查证] |
| S2 文档解析 | PDF/Word/图片 → Markdown | 提示用户配置文档解析工具或手动粘贴文本 |

### 4.5 用户配置文件

`.user-config.md`（不随插件分发）记录用户的能力槽配置与知识库注册。首次使用时由 `0-cold-start-interview` 引导生成；之后每次会话直接读取。

## 五、法律解释方法的运用：贯穿增强 + 素材分级

**根本立场**：法律解释方法（文义/体系/历史/目的/合宪/比较法）是论证的增强器，贯穿写作全程使用——不是"找不到权威依据时才启用的兜底"。即便已经从用户 KB、案例、学术资源获得了观点素材，仍要用方法对这些素材进行检验、整合与显性化。

**触发条件**：要件含义为不确定法律概念 / 一般条款，或本案事实涵摄存在歧义、双方有合理争议、司法实践有分歧、引用素材彼此冲突。

**素材分级**（为方法提供论据，而非替代方法）：

1. **用户已注册的解释 / 评注 KB**（脚注 [评注] / [释义]）——最权威
   - ↓ 实际调用并取得对应章节内容才标注；未命中即不留 KB 痕迹
2. **法律数据库案例检索 + 全网学术 / 实务搜索**（脚注 [案例] / [联网]）——通常都要执行
   - ↓ 必须实际调用并能定位段落 / 案号 / URL 后才标注
3. **方法论自助式解释**（依工具箱独立完成）
   不加任何脚注，正文展开方法步骤；严禁 [模型推理] / [此观点尚需查证] 占位脚注（真实来源原则）

无论调用到哪一级素材，解释段落都须显性展示方法在工作（文义边界 → 体系协调 → 历史 / 目的支撑 → 合宪 / 比较法校验，按需展开）。详见 workflow/3-drafting.md §3.4 与 methodology/legal-interpretation-toolkit.md。

## 六、关键约束（三层）

### 铁律层（违反即报告失败）

以下规则系分析正确性的底线保障，任何情况下不得违反、不得变通。

1. **法条校验铁律**：本 skill 在任何位置出现法条号或条文原文前，必须经 `[SLOT-LEGAL-DB]`（用户注册的法律数据库检索工具）实时校验，确认条文号正确、现行有效、内容一致后方可写入。模型对法条的训练记忆不得作为引用依据；用户知识库中的 OCR 文本可作为初稿参考来源，但最终输出以法律数据库返回原文为准。具体规则：
   - 校验颗粒度：同一条文在同一份报告中首次引用必须校验，后续引用可复用该次校验结果（不再重复调用）；
   - 校验失败处理：法律数据库返回 not_found / 已废止 / 内容差异时采用软标注——在该法条引用块或脚注中标注 `[⚠ 校验失败：<原因>，请人工核对]`，流程不阻断但留痕；
   - 降级策略：用户未配置 S1 法律数据库检索能力槽时允许降级，但全部法条引用必须统一带 `[⚠ 未经校验，请人工核对]` 标注；冷启动访谈应明示风险；
   - 校验记录留存：每次会话生成 `verification-log.md`（参 §七"法条校验留痕"），记录每个被引用条文的 ftnum、调用时间、返回的现行性、原文 hash，供 W4 差异核验回溯。

2. **方案先确认再开工 · 两闸独立**：完成 1-case-intake（W1）后必须用户确认才能进 W2；W2 拆为两道独立确认门——2a-preselection（候选请求权清单确认 = 闸一）与 2b-outline（解题大纲确认 = 闸二）必须分别用 AskUserQuestion 单独拍板，严禁合并为一次 AskUserQuestion（合并即视为方法论违例，与来源伪造同级）；闸二拍板后才能进 W3。

3. **阀门约束**：事实+设问 → 规范（内含于 W1/W2）：
   W1（案情解析与设问识别）只允许事实层+设问逐字+法律关系对：当事人/时间线/客观行为/数字/题干约定原话/各方主观状态的事实陈述/设问原文/X→Y法律关系对。禁止法律定性、条文号、争点预判、工作假设。
   W2（请求权预选与大纲）是法律层的真正起点：调用解题大纲生成引擎完成请求权预选、首次出现条文号、形成大纲，且每条文须经法律数据库双层校验。
   跨阀门污染（如 W1 出现条文号）一律视为方法论违例，应回到对应阶段重做。

### 规范层（标准遵循、偏离须说明理由）

以下规则构成报告质量的标准骨架，正常情况下应严格遵循；如因案情特殊需偏离，须在报告中简要说明理由。

4. **检视程式骨架**：报告正文须严格遵循三层四步内在结构（产生/未消灭/可行使）与五级外在检索顺序构建分析框架，并以 canonical-report-template.md 大纲填充骨架为展开基准。

5. **完整自含的脚注（真实来源原则）**：禁止"参前注 X"等前向引用；脚注仅用于实际调用并取得对应段落 / 页码 / 案号 / URL 的外部素材；自助式解释不加脚注，在正文展开方法步骤；严禁 [模型推理] / [此观点尚需查证] / 未实际调 KB 即标 [KB:] 等占位脚注。

6. **法律解释方法贯穿全程**：法律解释方法（文义/体系/历史/目的/合宪/比较法）是论证增强器而非兜底，在涵摄论证中须主动、显性地运用；即便已有权威素材支撑结论，仍应展示方法在工作。详见 §五。

7. **格式真相源**：examples/canonical-report-template.md 为报告格式的唯一权威基准——W3 写作启动前必读、W4 格式扫描以其为 diff 基准；format/output-format-spec.md 为辅助规范，两者冲突时以前者为准。

8. **论证深度比例原则（proportionality）**：论证篇幅须与要件争议程度成正比——明显成立/不成立者一句带过，存在合理争议者展开正反论证，核心争议要件须穷尽方法论工具。详见 methodology/proportionality.md。

9. **法条全文展开**：每个被引用的法条第一次出现必须以引用块完整展示原文（至少到"款"级别）。

10. **学术引用完整性**：
    生成的案例分析报告按学术规范在脚注中完整引用文献——书名、作者、出版社、年份、页码、刊名、卷期、URL 与访问日期等信息均应出现，确保学术诚信。引用格式见 methodology/gutachten-writing-rules.md §四。
    来自用户私有 KB 的引用采用"尽力字段 + 兜底字段"弹性规则：尽力提取作者 / 书名 / 出版社 / 年份 / 章节，缺失字段允许省略；兜底必须给出 ① KB skill 名称（[KB: <skill-name>]）和 ② 在 KB 中的章节 / 节级定位。

11. **客观第三人称**：避免"笔者认为""窃以为"等主观色彩；当事人称谓全文统一。

12. **三维核验闭环**：增量法条校验、大纲覆盖+事实忠实、格式扫描——核验完成后按 P1 → P2 → P3 优先级一次性修订，不循环。法条准确性维度强制比对 verification-log.md，无校验记录或为软标注者需用户确认。

### 弹性层（可在标注后调整）

以下要素可根据案情复杂度、用户指令或写作策略灵活调整，调整时在大纲阶段向用户说明即可。

13. **报告字数区间**：默认总字数 ≤ 30,000 字（含脚注）；案情简单时可压缩至 8,000–15,000 字，案情极复杂或用户明确要求时可突破上限至 40,000 字，但须在大纲确认时说明预估字数。

14. **反对说展开深度**：默认对每个核心争议要件展开正反两面论证；案情明朗、争议有限时可仅以一句话概括反对说立场而不逐要件展开。

15. **方法论自助式解释的路径选择**：法律解释工具箱提供文义 → 体系 → 历史 → 目的 → 合宪 → 比较法六阶位序，实际写作时可根据要件性质与可用素材选择最有说服力的路径组合，无需每次穷尽全部六阶。

16. **未详析请求权的排除理由表达方式**：对预选阶段排除的请求权基础，默认以脚注简述排除理由；亦可在大纲末尾集中列表说明，或在正文相关位置以括注形式处理。

### 六·一、阶段聚焦规则

各工作流阶段仅需关注与本阶段直接相关的规则子集，其余规则在该阶段暂不加载。

此规则不改变任何既有约束的效力——仅标示执行优先级，避免跨阶段注意力分散。

| 工作流阶段 | 聚焦规则 | 暂不关注 |
| --- | --- | --- |
| W1 案件解析 | 事实提取纪律、当事人识别、诉请类型判断、复杂度判定 | 格式规范、脚注、校验细则 |
| W2 预选+大纲 | 五级检索顺序、规范属性分类、解题大纲生成引擎全流程、法律数据库校验铁律、来源标注纪律 | 写作句式、proportionality、反对说深度 |

## 七、输出格式

**格式真相源**：examples/canonical-report-template.md——W3 写作启动前置门禁必读、W4 维度 3 格式扫描的 diff 基准。其中：

- §三 9 级标题精确矩阵：部分 / 一级黑体小三加粗（√×）/ 二级（一）大纲、（二）正文全角黑体四号 / 三级 1. 黑体小四 / 四级 (1) 半角宋体小四 / 五级 ① 宋体小四 / 六~九级加粗段头 A./a./(a)/(aa)；
- §四 大纲填充骨架双范式：侵权五分法（含规范属性）+ 违约四分法（含损害赔偿范围）+ 物权返还程式 + 不当得利程式；案由不同时由用户知识库检视程式或解题大纲生成引擎动态决定；
- §九 写作前置 14 项 checklist：W3 落笔门禁，任一项未勾不得开始撰写；
- §五 风格细节硬规则：法条简称、校验尾标三选一、脚注非硬覆盖率、目录强制、部分层条件化（多对当事人才加）、反对说位置统一、9 级深度允许差异。

**格式辅助规范**：format/output-format-spec.md——页面 / 字体 / 字号 / 行距 / 引用块视觉样式 / 字数上限 / Word 转换工具映射。两份文件冲突时以 canonical-report-template.md 为准。

要点摘要：

- A4 / 上下 2.54cm / 左 3.17cm / 右 3.07cm / 行距 1.25 倍；
- 正文宋体小四（12pt）；部分/一级黑体小三（15pt）/ 二级黑体四号（14pt）/ 三级黑体小四（12pt）/ 四~九级宋体小四（12pt）；引用块楷体_GB2312 小四（12pt）带左侧灰线；
- 段首 `  `（U+3000 全角空格 ×2）；
- 来源标识符四种且仅四种 `[评注]` / `[案例]` / `[联网]` / `[KB:]`（真实来源原则，脚注非硬覆盖率，可选不必填）；
- 必有目录页 `# 目录` + "（一）大纲""（二）正文"双 H2 外壳层；
- 部分层条件化：当事人对数 ≥ 2 加 `# 第X部分`，= 1 跳过；
- 禁出顶部凡例段、案情提要 / 设问独立段；
- 总字数 ≤ 30,000 字（含脚注）。

## 九、执行入口

收到触发请求时按下列顺序处理：

1. 检测 `.user-config.md` 是否存在：
   - 不存在 → 执行 `workflow/0-cold-start-interview.md`，访谈完成后保存配置；
   - 存在 → 跳过冷启动，进入第 2 步。

2. 解析用户输入：
   - 用户已提交案件材料 → 进入 `workflow/1-case-intake.md`；
   - 用户仅询问方法论 / 单段调用 → 直接读取相应方法论文件回应。

3. 按工作流逐段推进：W1（1-case-intake）→ W2 闸一（2a-preselection）→ W2 闸二（2b-outline）→ W3（3-drafting）→ W4（4-verification-and-output）。

4. 每段进入下一段前必须用户独立确认（除非用户在冷启动配置中显式启用"全自动模式"）；W2 两闸禁合并——闸一 AskUserQuestion 仅问"候选清单是否准确"，闸二 AskUserQuestion 才问"大纲是否准确 + 派生写作策略"。
