# 4-verification-and-output

工作流 4：校验、修订与交付

目的：对 W3 产出的 Markdown 草稿进行三维校验、一次性修订、用户预审、格式转换与最终交付，确保输出的鉴定式分析报告在法条准确性、大纲覆盖度、格式规范性三个维度上均达到交付标准。

## 一、三维校验（增量式、逐项通过）

三维校验按顺序执行，每一维度通过后方可进入下一维度。任一维度未通过即进入修订流程（见 §二），修订后重新从该维度开始校验。

### 1.1 维度一：增量法条校验

目标：确认正文中引用的每一条法条、司法解释条文均准确无误。

执行步骤：
1. 扫描合并草稿（`draft/merged.md`）中所有法条引用（正则匹配"《.+?》第\s*\d+\s*条"及司法解释引用模式）；
2. 逐条比对 `verification-log.md`：
   - 已标记 `[✓已校验]` 且在 W3 写作阶段完成校验的条目 → 直接通过；
   - 未出现在 `verification-log.md` 中的条目 → 立即通过 `[SLOT-LEGAL-DB]`（用户配置的法律数据库检索工具）查验原文，核对条号、款项、文字是否一致；
3. 司法解释条文：100% 强制重新校验，不承认 W3 阶段的校验结果——此类条文 OCR 差错率高、版本更新频繁，必须在交付前二次确认；
4. 校验结果追加写入 `verification-log.md`，标注 `[W4-✓]` 以区分 W3 阶段校验；
5. 发现错误时：
   - 文字性错误（错别字、款项编号错误）→ 直接在草稿中修正，标注 `[W4-修正]`；
   - 实质性错误（条文已废止、条文内容与引用意图不符）→ 标记为修订项，进入 §二 修订流程。

通过标准：`verification-log.md` 中每一条引用均有对应的校验标记（`[✓已校验]` 或 `[W4-✓]`），且无未解决的实质性错误。

### 1.2 维度二：大纲覆盖度 + 事实忠实度

目标：确认正文完整覆盖了 W2 最终大纲的全部要件，且所有事实陈述均与原始案情材料一致。

**大纲覆盖度检查：**
1. 读取 W2 闸二拍板的最终大纲（`outline/final-outline.md`）；
2. 逐项核对大纲中的每个要件是否在正文中有对应的论证段落；
3. 检查要件的展开深度是否符合比例原则分级：
   - "明显"要件 → 是否有裁判式简要涵摄；
   - "争议"要件 → 是否有鉴定式分析；
   - "核心争议"要件 → 是否有完整鉴定式 + 法律解释方法展开。
4. 缺失要件或深度不足 → 标记为修订项。

**事实忠实度检查：**
1. 正文中引用的案件事实（时间、金额、合同条款、当事人行为等）与 W1 确认的事实梳理材料逐项比对；
2. 不得出现"推测性事实"——即正文中出现但原始材料中无依据的事实陈述；
3. 当事人称谓与 W1 确认的当事人表完全一致；
4. 发现不一致 → 标记为修订项。

通过标准：大纲要件 100% 覆盖，事实陈述 100% 可溯源。

### 1.3 维度三：格式扫描

目标：确认草稿格式完全符合 `examples/canonical-report-template.md` 和 `format/output-format-spec.md` 的规范。

启动一个 general-purpose subagent 做逐项 diff 格式检查，基准为 `examples/canonical-report-template.md` §三 9 级标题矩阵 + §五 风格细节硬规则 + §九 14 项 checklist。

扫描矩阵（14 项硬规则，每项给出"通过 / 漂移 + 行号 + 修复建议"）：

1. 是否含 `# 目录` 段（强制目录页）？
2. 部分层判断：当事人对数 ≥ 2 时是否以 `# 第X部分`（H1）起首；= 1 时是否跳过部分层？
3. 一级是否含 `## （一）大纲` 与 `## （二）正文` 双 H2 外壳层（全角括号、黑体小三加粗，强制）？
4. （一）大纲内是否含"请求权基础预选结果"与"检视程式概览"两个子节？
5. 二级（正文内）是否用 `### 1. / 2. / 3.` 三层四步主层级（H3、黑体四号加粗）？
6. 三级是否用 `#### (1) / (2) / (3)` 要件展开层（H4、半角圆括号、黑体小四加粗）？
7. 四级是否用 `##### ① / ② / ③` 子要件层（H5、宋体小四不加粗）？
8. 五级是否用 `###### A. / B. / C.` 细分论证层（H6、宋体小四不加粗）？
9. 6~9 级是否以加粗段头模拟 **a.** **(a)** **(aa)** **(aaa)**（半角，宋体小四，仅段头加粗）？
10. 所有正文段首是否以 `  `（U+3000 全角空格 ×2）开头？引用块、标题、加粗段头自身不缩进？
11. 法条引用块是否用 `> 《民法典》第XX条：…` 格式，校验尾标三选一（无尾标 / ⚠校验失败 / ⚠未经校验）？
12. 反对说位置同一报告内是否统一（在对应要件的抗辩子层级下，位置一致）？
13. 真实来源原则：脚注（如有）是否仅用 `[评注]/[案例]/[联网]/[KB:]` 四种来源标识？是否禁用 `[模型推理]` `[此观点尚需查证]` 等占位脚注？
14. 任一阶段不成立后续是否标"无须检视"而非空缺？

通用项（保留）：
- 法律术语使用一致性（参 gutachten-writing-rules.md）
- 当事人称谓全文统一
- 引用格式：《法规名》第 XX 条（书名号 + 空格）
- 不含外语方法论标签
- 各部分间 `---` 分隔
- 总字数 ≤ 30,000 字

门禁：14 项扫描矩阵任一项命中"漂移" → 视为 P1 优先级（与法条准确性同级）→ 强制返回 W3 修复，禁止进 Word 转换。

逐项 diff 操作：subagent 须以 `examples/canonical-report-template.md` 各节标题为索引，对草稿做"项 → 节 → 行号 → 通过/漂移 + 修复建议"四列输出。

输出"格式扫描报告"，含 14 项扫描矩阵结果表 + 通用项清单 + 整体判定（pass / fail）。

## 二、一次性修订（禁止循环）

修订纪律：W4 的修订是"一次性"的——收集三维校验的全部修订项后，集中进行一次修订，修订完成后不再重新执行完整校验流程。

### 2.1 修订项汇总

将三维校验中标记的所有修订项汇总至 `revision-items.md`，按以下分类排列：
- **法条类修订**：条文修正、条文替换、条文补充；
- **内容类修订**：要件补充论证、事实修正、深度调整；
- **格式类修订**：标题层级调整、引用格式统一、脚注重编号。

### 2.2 集中修订执行

按修订项清单逐项在草稿上完成修改：
- 法条类修订：修正后须再次通过 `[SLOT-LEGAL-DB]` 校验修正后的条文；
- 内容类修订：补充的论证段落须遵守 W3 的写作规范（比例原则分级、法律解释方法等）；
- 格式类修订：逐项对照 checklist 完成调整。

### 2.3 修订后快速复核

修订完成后，仅对修订项进行快速复核（非完整三维校验）：
- 修订的法条是否已校验通过；
- 修订的段落是否符合比例原则深度要求；
- 修订后的格式是否通过对应 checklist 项。

复核通过 → 进入 §三 用户预审。
复核发现新问题 → 就地修正，不再开启第二轮修订循环。

## 三、用户预审

在最终格式转换前，将修订后的 Markdown 草稿提交给用户进行预审。

### 3.1 预审材料准备

向用户呈现以下材料：
1. `draft/merged.md`（修订后的完整草稿）；
2. `verification-log.md`（法条校验日志摘要）；
3. `revision-items.md`（修订项清单及处理结果）；
4. 三维校验的通过情况摘要。

### 3.2 用户反馈处理

根据用户反馈进行分类处理：
- **无需修改** → 直接进入 §四 最终格式输出；
- **小幅调整**（文字修改、格式微调、个别表述调整）→ 直接在草稿上修改，修改后进入 §四；
- **实质性修改**（要件重新分析、请求权变更、新增法律关系对）→ 回退至对应工作流阶段（W2 或 W3），不继续在 W4 内处理。

### 3.3 预审通过确认

用户明确确认"通过"或"可以输出"后，进入 §四。未经用户确认不得进行最终格式转换。

## 四、最终格式输出

用户预审通过后，将 Markdown 草稿转换为用户所需的最终交付格式。

### 4.1 输出格式确认

读取 `.user-config.md` 中的输出格式偏好配置：
- 若用户指定 Word（.docx）格式 → 通过 `[SLOT-WORD-OUTPUT]`（用户配置的文档转换工具）进行转换；
- 若用户指定 PDF 格式 → 先转换为 Word，再由用户自行转换为 PDF（或直接通过 `[SLOT-WORD-OUTPUT]` 输出后转换）；
- 若用户仅需 Markdown → 跳过格式转换，直接进入 §五 交付。

### 4.2 Word 格式转换规范

通过 `[SLOT-WORD-OUTPUT]` 转换时，须确保以下格式参数符合 `format/output-format-spec.md` 的规范：
- **字体**：正文宋体 / 仿宋，标题黑体（具体字号见规范文件）；
- **行距**：1.25 倍（与 `format/output-format-spec.md` 一致）；
- **页边距**：按规范文件设定；
- **页眉 / 页脚**：按规范文件设定（如有）；
- **目录**：自动生成，层级与 Markdown 标题对应。

### 4.3 转换后检查

转换完成后，对输出文件进行快速检查：
- 文件是否可正常打开；
- 标题层级是否正确映射；
- 脚注是否正确显示；
- 中文字符是否无乱码；
- 表格（如有）是否正确渲染。

发现问题 → 调整转换参数后重新转换，最多重试一次。

## 五、交付

### 5.1 交付文件清单

最终向用户交付以下文件：
1. **正文文件**：`output/鉴定式分析报告.docx`（或 `.md` / `.pdf`，取决于用户偏好）；
2. **校验日志**：`output/verification-log.md`（法条校验记录，供用户存档备查）；
3. **修订记录**：`output/revision-items.md`（修订项清单，供用户了解定稿过程）。

### 5.2 交付说明

交付时向用户简要说明：
- 报告中引用了多少条法条，校验通过率；
- 大纲覆盖度（是否 100% 覆盖 W2 最终大纲的全部要件）；
- 法律解释方法的使用情况（多少个要件进行了完整解释方法展开）；
- 如有需要用户后续自行确认的事项（如时效问题、证据充分性等），在此一并提示。

## 六、副产物保留

- 案情解析摘要 → `parsed/case-summary.md`
- 解题大纲 → `outline/final-outline.md`
- 各 subagent 草稿 → `draft/part-N.md`
- 法条校验台账 → `verification-log.md`
- 格式门禁记录 → `format-precheck.md`
- Markdown 原稿 → `output/<案件简称>-鉴定式分析报告.md`（或 `鉴定式分析报告.md`）
- Word 终稿 → `output/<案件简称>-鉴定式分析报告.docx`（或 `鉴定式分析报告.docx`）
