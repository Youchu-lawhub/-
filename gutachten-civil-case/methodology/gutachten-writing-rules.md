# gutachten-writing-rules

鉴定式写作规范

本文件汇总鉴定式案例分析的语言风格、术语用法、引用规范、标点格式等具体写作约定。

## 一、语言风格

### 鉴定式句式（用于争议/解释要件）

- **大前提（假设语气）**：使用"若……，则……""倘……则……""须满足……方能成立……"等表达。
- **小前提（涵摄事实）**：使用"本案中，……""依本案查明事实，……""就本案而言，……"等表达。
- **结论（陈述/确定语气）**：使用"故……成立""综上，要件……不满足""因此……请求权不能成立"等表达。

### 裁判式句式（用于明显要件、程序事项）

直接表达结论，简洁。例：

> "本案中，A 与 B 签订书面买卖合同（证据 1），合同关系成立。"

### 段落结构

- 首句开门见山，直接表达本段所欲论证的命题；
- 中段展开论证，原则上一段一论点；
- 段落末句作小结。

## 二、术语统一

### 当事人称谓

- 使用案情提要中给定的姓名 / 简称（如"甲"、"乙"），全文统一；
- 不混用"原告 / 被告"与"当事人姓名"——选定一种贯穿到底（推荐使用案情中的姓名/字母简称，使报告与诉讼立场无关）。

### 请求权 / 规范基础术语

- 给付之诉 → 用"请求权基础"；
- 确认之诉、形成之诉、非讼程序 → 用"规范基础"；
- 不混用。

### 检视层级术语

固定使用：

- "请求权（已）产生"
- "请求权（未）消灭"
- "请求权（可）行使"
- "构成要件 / 责任成立要件 / 产生要件"
- "产生抗辩 / 消灭事由 / 停止性抗辩"

## 三、法条引用规范

### 三·〇 校验前置

任何法条号、法条原文、款项序号写入正式草稿前，必须有法律数据库校验记录。这是校验铁律的写作端落地：

- **首次必校验**：每条法条第一次写入草稿前，先查 W3 阶段产出的 verification-log.md：
  - 已记录且 [✓已校验] → 直接复用；
  - 未记录 → 立即调用`[SLOT-LEGAL-DB]`（用户注册的法律数据库检索工具），取条文原文 + 现行有效性，把校验记录补写入 verification-log.md，再写入草稿；
  - 校验失败 → 在引用末尾加软标注 [⚠ 校验失败]，记录原因到 verification-log.md，不阻断写作。
- **后续可复用**：同一条文在同一份报告中第二次及以后引用时，不重复调用。
- **以法律数据库为准**：当法律数据库返回原文与知识库 OCR 文本不一致，一律以法律数据库返回为准写入草稿；差异记到 verification-log.md 备注列。
- **模型记忆禁用**：基于训练记忆"我记得这条是这么写的"直接落笔，是**严令禁止的**——任何条文号必须先有校验记录。
- **降级模式**：用户机器未配置`[SLOT-LEGAL-DB]`时，所有法条引用统一在末尾加 [⚠ 未经法律数据库校验，请人工核对]；流程不阻断，但 W4 维度 1 会列出全部软标注交用户决定。

### 引用格式

- **首次引用**：民法典第 577 条（或简称 + 条文号）；
- 如使用了简称，简称需在凡例或脚注中第一次出现时给出全称；
- **同条多次引用**：可使用"该条""前引第 577 条"。

### 必须展开法条原文

每一个被引用的法条（包括司法解释条款），第一次出现时必须以引用块（blockquote）展示完整原文，至少到"款"级别，并附校验标记：

```markdown
> 《民法典》第 577 条：当事人一方不履行合同义务或者履行合同义务不符合约定的，应当承担继续履行、采取补救措施或者赔偿损失等违约责任。 [✓ 已校验 2026-05-28，现行有效]
```

后续引用同一条文，可直接使用条文号无须再展开，亦无须再标校验标记。

### 多条文集中引用

如需引用多个法条建立请求权基础，按"主条文 + 配套条文"分层展开，每条独立成段。

## 四、脚注规范

### 脚注格式

- 每注完整、自含信息——不允许"参前注 X""同前注 Y"等前向引用；
- 首次引用书目：作者. 书名（出版社, 年份）, 第 X 页；
- 首次引用论文：作者:《篇名》，载《刊名》XXXX 年第 X 期，第 X 页；
- 首次引用网络资料：作者/网站. 篇名. URL（访问日期）。

### 来自用户私有知识库的引用（页码/出版信息可能缺失）

用户注册的解释 / 评注类 KB 常基于 OCR 处理过的电子书或用户自制摘录，页码、边码、出版社、出版年、版次等字段经常缺失。脚注组装时按以下"尽力字段 + 兜底字段"规则处理：

**尽力提取（按优先级）**：

1. 作者（如能在 KB 文档头/前言中识别）；
2. 书名 / 篇名（KB 文档标题、前言或章节标题中的原书名）；
3. 出版社、出版年、版次（如 KB 中可见）；
4. 章节标题或所在节名 / 节号（OCR 资料一般保留章节结构，应当作为页码替代物使用）；
5. 页码 / 边码（如有则给出）。

**兜底字段（至少要有的两项）**：

1. KB skill 名称（例如 [KB: <用户已注册的评注类KB>]）；
2. 在 KB 中的检索路径或章节定位（如"第三编合同·第十一章 / 第577条注释"）。

**示例**（页码缺失）：

> 作者. 民法典合同编评注，第十一章·第577条注释（出版社, 2023） [KB: <评注KB名称>]

**示例**（出版年与出版社缺失）：

> 作者. 民法学教程，第二章第三节"意思表示" [KB: legal-kb]

**示例**（仅有 KB 与章节定位）：

> 民法分则评注·物权编 / 第236条 第三段评注 [KB: legal-kb-agentic]

只要"KB 名称 + 章节定位"两项齐全，即视为合格脚注。书名 / 作者 / 出版社 / 年份等字段按"有则填、无则省"原则处理，不强求齐全。

### 命题来源分类与可信度（六类体系）

报告中所有实质性命题须归入以下六类之一，以便读者独立判断证据等级：

| 类别 | 说明 | 标注方式 |
| --- | --- | --- |
| 实定法引用 | 经法律数据库校验的现行有效法条 | 条文号 + [已校验] / [校验失败-见附注] |
| 司法解释/指导意见 | 经法律数据库校验的规范性文件 | 条文简称 + 条号 + 校验状态 |
| 权威评注引用 | 来自用户知识库的评注内容 | [KB: <KB名称>] + 章节定位 |
| 案例引用 | 来自司法判例 | [案例] + 案号 |
| 方法论自主分析 | 分析者依据方法论工具独立得出的解释意见 | 正文展开方法步骤（无脚注） |
| 模型知识推断 | 基于模型训练数据的一般性法律知识 | 仅限正文论证中的逻辑辅助，严禁以脚注形式出现，严禁单独支撑结论 |

**可信度层级**：实定法/司法解释（已校验）可直接作为裁判依据 → 权威评注/案例/方法论自主分析可作为论证增强 → 模型知识推断不可单独支撑结论。

**禁止行为**：将模型记忆包装为"通说认为""实务中法院一般认为"；将用户知识库 OCR 文本未经法律数据库校验作为法条原文；省略来源类别使读者无法判断可靠性；混合使用两种来源仅标注其中一种。

### 脚注层级（真实来源原则，可选不必填）

脚注仅当实际调用并取得对应段落 / 页码 / 案号 / URL 的外部素材时才加；不强制覆盖率：

- 学理观点 / 评注观点 / 司法解释释义（已实际取得） → 加脚注
- 法条原文 / 公开判例 → 在正文中标明出处（条文号 / 案号），可不必脚注
- 自助式方法论解释（文义/体系/历史/目的/合宪/比较法）→ 不加脚注，正文展开方法步骤构成密度

**严禁**：[模型推理] [此观点尚需查证] [释义]（已并入 [评注]）/ 未实际调 KB 即标 [KB:] 等占位脚注

### 来源标识符（四种且仅四种）

- `[评注]` —— 来自权威条文评注（含释义类）
- `[案例]` —— 来自司法判例
- `[联网]` —— 来自联网检索文章
- `[KB: <KB名称>]` —— 来自用户注册的知识库

## 五、标题层级（与 canonical-report-template 统一）

写作全程使用 canonical-report-template.md §三 定义的 9 级标题体系，不得另立层级：

| 层级 | Markdown 标记 | 用途 |
| --- | --- | --- |
| 部分 | `# 第X部分` | 多对当事人分组（当事人对数 = 1 时跳过） |
| 一级 | `## （一）大纲` / `## （二）正文` | 外壳层（H2） |
| 二级 | `### 1. 请求权已产生` | 三层四步主层级（H3） |
| 三级 | `#### (1) 合同成立` | 要件展开层（H4） |
| 四级 | `##### ① 要约` | 子要件层（H5） |
| 五级 | `###### A. 构成要件` | 细分论证层（H6） |
| 六~九级 | `**a.**` `**(a)**` `**(aa)**` `**(aaa)**` | 段头加粗（非 heading） |

**注意**：大纲部分（`## （一）大纲`）内部的子节（如"请求权基础预选结果""检视程式概览"）使用 H3（`###`），与正文部分的三层四步标题层级相同但内容不同。

## 六、引用块与表格

- **法条原文** —— 用 Markdown 引用块（> ...）；
- **比对要素 / 类型化对照** —— 用 Markdown 表格；
- **流程示意** —— 用编号列表，必要时配合代码块呈现。

## 七、避免的表达

- "显然""不言自明""毋庸赘言"——回避性论证，禁用；
- "笔者认为""窃以为""个人观点"——鉴定式中应客观第三人称，避免主观色彩；
- "百分之百""绝对"——除引用法条外，慎用绝对化表达。

## 八、长度参考（含脚注）

完整鉴定式报告的合理长度：

- 简单单一请求权 → 4,000–8,000 字；
- 中等复杂（2–3 法律关系对）→ 8,000–18,000 字；
- 复杂多方多请求权 → 18,000–30,000 字。

超过 30,000 字时应当反思是否过度展开非核心要件，参考 proportionality.md 删减。

## 九、常见偏差清单

以下为 W3 写作阶段高频出现的质量偏差，模型在草稿完成后应逐项自查：

### 结构类偏差

- **跳步涵摄**：列出构成要件名称后直接给结论，未完成"定义→涵摄→小结论"三步完整链。
- **抗辩位错**：将否认（反驳要件事实）误归入抗辩段，或将抗辩事由混入产生要件检视。
- **骨架错配**：对侵权请求权使用了违约检视程式的骨架（如缺少"不法性"层），或对违约请求权使用了侵权五分法（如多出"规范属性"层）。
- **三层截而不断**：请求权产生已否定后仍展开消灭和行使层检视（正确做法：明确说明"鉴于请求权未产生，以下两层无需检视"后终止）。

### 论证类偏差

- **假设句缺失**：鉴定式分析开头未以"X 或可向 Y 依据……请求……"假设性结论引入。
- **法条浮引**：行文中写"依第XX条"但该条文既无引用块展示原文、亦无校验记录。
- **来源伪装**：将模型训练记忆包装为"通说认为……"或"实务中法院一般……"，未附任何来源标注。
- **循环定义**：用同义词替代构成要件定义（如"过失即未尽注意义务"后立刻又用"未尽注意义务即过失"）。

### 格式类偏差

- **层级错配**：四级标题 (1) 使用了全角括号、五级标题 ① 加了粗体。
- **段首缩进遗漏**：正文段落未以全角空格（U+3000 ×2）开头。
- **引用块校验尾标缺失**：法条引用块末尾缺少（已校验）或（⚠ 校验失败）状态标记。
- **脚注前向引用**：出现"参前注 X""同前注 Y"等非自含脚注。

**自查时机**：每完成一个请求权基础的正文写作后即刻逐项扫描，不等全篇完成后再批量修正。
